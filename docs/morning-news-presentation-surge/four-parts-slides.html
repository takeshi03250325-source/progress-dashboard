新聞記事の自動化ツール

説明するときの「4つのパーツ」で整理する

トリガー → ソース元 → 処理する場所 → 届ける先

文字は大きめ・配色は明るめ（発表・投影向け）

4つのパーツ（ひと目の図）

左から順にデータが流れるイメージ

→ → →
1

トリガー

いつ・何をきっかけに動くかを決める

2

ソース元

データがある場所を決める

3

処理する場所

データを加工してレポートにする場所を決める

4

届ける先

完成したレポートを届ける先を決める

PART 1

トリガー

いつ・何をきっかけに動くかを決める

このプロジェクトでの例

  • 定時：GitHub Actions のスケジュール（毎朝・日本時間 6:30 前後のイメージ）
  • 手動：Actions の「このワークフローを実行」（workflow_dispatch）
  • 任意：LINE で「NG」と返したとき Webhook が再取得のきっかけに（開発・検証時）

PART 2

ソース元

データがある場所を決める

このプロジェクトでの例

  • RSS：各メディアのフィード（設定は YAML などで管理）
  • Google ニュース：検索ワード（例：建設業×日本）で記事 URL を取得
  • 設定ファイル：どのソースを何件読むか・絞り込み条件をコードと一緒にバージョン管理

PART 3

処理する場所

データを加工してレポートにする場所を決める

このプロジェクトでの例

  • 実行環境：GitHub Actions 上の Linux（npm / Node でスクリプト実行）
  • 加工内容：記事の選定・重複除外・AI（Gemini）で X 投稿案の文案生成
  • 出力物：当日の候補リスト・投稿履歴 JSON（リポジトリに保存される）

PART 4

届ける先

完成したレポートを届ける先を決める

このプロジェクトでの例

  • LINE プッシュ：Messaging API で指定ユーザーにテキストを分割送信
  • 中身：本日の候補・X 投稿案・出典などをまとめて届ける
  • Secrets：チャネルのアクセストークン・届け先ユーザー ID は GitHub に安全に保管

一周まわすとこうなる

① トリガー … 毎朝の定時・手動・（任意）Webhook
▼
② ソース元 … RSS・Google ニュース・YAML 設定
▼
③ 処理 … GitHub Actions で取得・選定・AI 文案・JSON 保存
▼
④ 届ける先 … LINE にプッシュ（自分のスマホで確認）

説明するときは、この 4 語を口ぐせにすると聞き手が迷いにくいです。
