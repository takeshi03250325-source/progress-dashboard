推奨する作り方（図解）

朝の建設ニュース → 1本選ぶ → X用の文 → LINEで「OK？」→ OKならX投稿、という仕組みを 分かりやすい順番で作るときの目安です。

運用の切り分け（はっきり書いておく）

プログラムに任せる／人が担う

自動 プログラムに任せた方がよいこと

  • 定時起動 — 毎朝など、決まった時刻に処理を始める（cron / GitHub Actions など）。
  • データ取得と加工 — RSS 等の取得、候補の 1 本への絞り込み、X 用文案の生成、いったんの保存。
  • API 呼び出しの実行 — LINE へのプッシュ、承認後の X 投稿 API の実行（ロジックとトークンはコード・Secrets 側）。
  • 失敗時の通知・再実行 — エラー検知、Slack 等への通知、ジョブのリトライや手動再実行の入口。
  • このページの運用も — 更新後の Surge デプロイはコマンドで繰り返す。資料用の画面キャプチャはリポジトリの npm run screenshot:diagrams（Playwright）に任せると Mermaid まで含めて再現しやすい。

人 人が担う方がよいこと

  • 何を見せて何を承認するかの設計 — 静的 HTML（Tailwind）で「画面の見本」を決める。LINE に送る文面の前提もここで固まる。
  • 公開してよいかの判断 — LINE での「OK」／NG。自動投稿の最終ゲートは人が持つ。
  • ルールとポリシー — どのソースから取るか、X API・利用規約の確認、チーム内の運用ルール。
  • 初回のインフラ — Surge ドメインの確保、各サービスのアカウント作成、Secrets の登録（作業そのものは一度人手で行う）。
  • このページの中身 — 図の文言・Mermaid・デザインの更新は index.html を編集する（創作は人）。

迷ったら「繰り返し・同じ手順・ミスが痛い」はプログラム、「意味・責任・初回の決めごと」は人、と考えると整理しやすいです。

まずこれだけ（4ステップ）

プレビューしなくても読めるように、文字だけの図にしています。

  STEP 1          STEP 2              STEP 3              STEP 4
 ┌─────────┐    ┌─────────────┐    ┌──────────────┐    ┌─────────────┐
 │ 画面の   │ → │ 記事の取得   │ → │ LINE と X を  │ → │ 失敗したら   │
 │ 見本を   │   │ と「1本選ぶ」 │   │ つなぐ        │   │ 気づける仕組み│
 │ 先に作る │   │ ロジック     │   │               │   │              │
 └─────────┘    └─────────────┘    └──────────────┘    └─────────────┘
   HTML            プログラム         API連携            通知・再実行
  (Tailwind)      (RSSなど)         (公式の仕組み)      (運用)
ステップ やること こうしておくと楽
1 完成イメージの 静的HTML（Tailwind）を作る 「LINEに何を見せるか」「何を承認するか」が決まる
2 ニュース取得（RSS等）→ 1本に絞る → X用の文 → いったん保存 まだLINE/Xに繋がなくてよい
3 LINEで候補を送る → 返信で OK → X APIで投稿 人が最終チェックしてから公開
4 エラー通知・APIキーの管理・必要なら再実行 朝の1回だけに頼らない

覚え方: 「見た目 → 中身 → 外との配線 → 運用」の順。

⑥ で NG のときは、投稿せず別処理（再提案・翌日まで保留など）にできます。

なぜ「HTMLを先」なのか

   いきなりプログラムだけ作る        先にHTMLで見本を作る
   ─────────────────────          ─────────────────────
   あとから「画面が違う」            最初に「こう見せたい」が固定できる
   手戻りが増えやすい                そのあとデータを流し込むだけに近づく

注意（短く）

  • 記事は RSS・公式の配信など、ルールがはっきりした取り方を優先するのが安全です。
  • X への投稿は API の条件・料金を事前に確認してください。

Surge で公開する手順

このフォルダだけをアップロードすればOKです（ビルド不要）。

  1. ターミナルでこのフォルダへ移動: cd docs/recommended-build-flow-surge
  2. 初回のみ: npx surge login
  3. 公開（ドメインを確実に合わせる）: npx surge . ads-recommended-build-flow.surge.sh

公開後のURL: https://ads-recommended-build-flow.surge.sh

別リポジトリで実装するときは、このフォルダごとコピーして使って構いません。
